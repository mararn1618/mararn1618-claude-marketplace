# agent-kb -- Format Reference

This file is the single source of truth for the knowledge base output format. The build operation reads this before producing any files.

---

## Output Structure

A knowledge base is a folder tree:

```
<target-path>/
  CLAUDE.md              # Root routing file (for Claude Code)
  AGENTS.md              # Root routing file (for Codex / other agent frameworks)
  01-<domain>/
    CLAUDE.md
    AGENTS.md
    article-a.md
    article-b.md
  02-<domain>/
    CLAUDE.md
    AGENTS.md
    article-c.md
  ...
```

Folders are numbered with zero-padded two-digit prefixes (`01-`, `02-`, ...). The number defines the recommended reading order. Later folders may assume context from earlier ones.

---

## Dual Routing Files: CLAUDE.md + AGENTS.md

Every folder in the KB gets **two** routing files with **identical content**: `CLAUDE.md` (read by Claude Code) and `AGENTS.md` (read by Codex and other agent frameworks). Always write both. Never let them diverge.

### Root Routing File Format

The root `CLAUDE.md` / `AGENTS.md` at `<target-path>/`:

````markdown
---
scope: <one-line description of the entire KB's domain>
description: <what an agent finds here and when it should navigate in, specific enough to judge relevance from this text alone>
---

# <KB Title>

<2-4 sentences: what this KB is, who the primary reader is, what it covers. Mention the folder numbering = recommended reading order.>

## How to navigate

1. Start with the first numbered folder for orientation.
2. Each folder has its own `CLAUDE.md` / `AGENTS.md` with a Contents table.
3. Articles are self-contained: readable without loading the entire KB.

## Contents

| Name | Type | Description | Updated |
|------|------|-------------|---------|
| [01-orientation/](01-orientation/) | folder | <what this folder covers> | <YYYY-MM-DD> |
| [02-business/](02-business/) | folder | <what this folder covers> | <YYYY-MM-DD> |
````

### Subfolder Routing File Format

Each numbered domain folder's `CLAUDE.md` / `AGENTS.md`:

````markdown
---
scope: <what domain this folder covers>
description: <one-line description specific enough for relevance matching>
---

# <NN> <Domain Title>

<2-3 sentences: what lives in this folder and when an agent should read it. Be specific about the topics covered so an agent can judge relevance from this text alone.>

## Contents

| Name | Type | Description | Updated |
|------|------|-------------|---------|
| [article-name.md](article-name.md) | context | <what the agent finds in this file, under 80 chars> | <YYYY-MM-DD> |
````

### Contents Table Columns

| Column | Description |
|--------|-------------|
| **Name** | Relative markdown link for files/folders. |
| **Type** | `context` for markdown articles, `folder` for subdirectories. |
| **Description** | What an agent finds here. Drives relevance matching. Under 80 characters. |
| **Updated** | Date the content was last written (`YYYY-MM-DD`). |

## Article Format

Each article is a markdown file in a domain folder. Articles are **prose-first and pointer-rich**: written so a human or agent can read them standalone without loading the rest of the KB.

### Frontmatter

Every article opens with YAML frontmatter:

```markdown
---
title: <article title>
scope: <what this article covers and when an agent should read it>
updated: <YYYY-MM-DD>
---
```

### Body

- Open with a **one-line framing** sentence that states what this article is about.
- Use clear headings (H2, H3) to structure sections.
- Cross-link to other articles in the KB with relative paths: `[link text](../04-other-domain/related-article.md)`.
- Cross-link to source material when possible: `Source: <path-to-original>`.
- Tables are welcome for structured comparisons.
- No watermarks, no "generated by AI" footers, no version metadata.

### Quality Bar

An article is done when:
1. A cold reader (human or agent) can understand it without reading other articles first.
2. Claims are traceable to a source (original file path, meeting date, document name).
3. It compresses the source material rather than copying it verbatim.
4. It uses domain vocabulary consistently (defined in a glossary article if needed).

---

## Folder Naming Convention

- Numbered: `01-orientation`, `02-business-context`, `03-functional-scope`
- Lowercase kebab-case after the number prefix.
- The first folder should always be orientation/overview material.
- Folder count: aim for 5-15 domains. Fewer than 5 means the clustering is too coarse; more than 15 means too granular.

---

## Article Naming Convention

- Lowercase kebab-case: `vdm-overview.md`, `identity-propagation-options.md`
- No number prefixes on articles (only folders are numbered).
- Name describes content, not type: `stakeholder-map.md` not `reference-01.md`.

## Sensitive Data Categories (for scrub phase)

The scrub phase scans for these categories by default. The skill should also brainstorm additional categories based on the KB's domain and confirm with the user before scanning.

| Category | Examples | Pattern hints |
|----------|----------|---------------|
| **Filesystem paths** | `/Users/john/Documents/...`, `C:\Users\...`, `~/personal/...` | Absolute paths with usernames, home dirs |
| **Internal URLs** | Confluence, Jira, SharePoint, intranet links, localhost URLs | Domain patterns: `*.atlassian.net`, `*.sharepoint.com`, internal hostnames |
| **Person names** | Full names of individuals in meeting notes, stakeholder lists | First + last name patterns, email local parts |
| **Credentials** | API keys, tokens, passwords, connection strings | `Bearer ...`, `sk-...`, `password=`, base64 blobs |
| **Email addresses** | Personal or corporate email addresses | `*@*.*` patterns |
| **IP addresses** | Internal IPs, server addresses | `10.*`, `192.168.*`, `172.16-31.*`, specific public IPs |
| **Phone numbers** | Direct lines, mobile numbers | Digit sequences with country codes or area codes |
| **Proprietary identifiers** | Customer IDs, account numbers, internal ticket IDs | Context-dependent, varies by domain |

The scrub report flags findings. It does **not** auto-redact. The user reviews and decides what to redact, keep, or generalize.
